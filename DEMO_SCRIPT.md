## The narrative spine (memorize this — everything else is dressing)

> "Wildfires kill people partly because dispatch decisions get made on incomplete data and AI assistants can hallucinate evacuation guidance that gets people killed. Wildfire Watch ingests real fire feeds, predicts spread with a physics-grounded ML model, and routes every recommendation through a safety gate that writes a tamper-evident audit row **before** any alert is sent. If the model is uncertain, a human approves. If the AI says something unsafe, Guardrails blocks it. Every decision is replayable from the chain."

Three words to land: **predict, gate, audit.**

## Beat 3 — The safety gate (1:30 → 2:45) "why the audit row matters"

This is the AI Safety track money beat. Slow down here.

**Switch tab:** open `functions/alert/safety_gate.py` in the editor.

**Say:**
> "Every advisory passes through this Lambda. The order of operations is the safety contract — six steps, in this exact order:"

**Walk through (don't read line-by-line — point at each):**
1. Validate input shape
2. Generate advisory via Bedrock
3. **Append the prediction row to the audit hash chain** ← stop here
4. Validate via Bedrock Guardrails
5. Append a `guardrails_outcome` row linked to the prediction
6. Confidence threshold check — if < 0.65, route to human review

**Land the punchline:**
> "Step 3 is the contract. The audit row is committed to DynamoDB **before** Guardrails even runs. That means if the AI generates something dangerous and Guardrails catches it, we still have a permanent record of what was generated and who decided it shouldn't go out. We can't quietly retry until we get an answer we like."

**Switch tab:** open the `wildfire-watch-audit` DynamoDB table in the AWS console.

**Click:** "Explore items" → filter by a `fire_id` from the demo. Show the rows in chronological order.

**Audience sees:** Multiple rows per fire — each with `prediction_id`, `prev_hash`, `record_hash`, `event` field showing `prediction` → `guardrails_outcome` → (`alert_sent` if approved). Each `prev_hash` matches the previous row's `record_hash`.

**Drop in:**
> "SHA-256 hash chain. Each row commits the hash of the previous row for the same fire. Tamper with any middle row and the next one's hash stops matching. We pivoted to this from QLDB because AWS isn't accepting new QLDB ledgers anymore — but the immutability property is preserved."

**Fallback:** If the AWS console is slow, point at `functions/alert/audit.py` lines 30-100 in the editor — the `_canonical_hash`, `log_prediction`, and `verify_chain` functions tell the same story in 70 lines of Python.

## Beat 4 — The three exit paths (2:45 → 3:30) "the gate has teeth"

**Open the Step Functions console** in the second tab → `wildfire-watch-safety` state machine → "Definition" tab.

**Audience sees:** the visual graph — `SafetyGate` → `ActionRouter` Choice node with three branches.

**Say:**
> "The safety gate returns one of three actions. Step Functions routes accordingly:"

Point at each branch:
1. **APPROVED** → AlertSender. Confidence is high, Guardrails passed. SMS goes out.
2. **HUMAN_REVIEW_REQUIRED** → NotifyDispatcherAndWait. Confidence below 0.65. Step Functions pauses for up to 5 minutes waiting for the dispatcher to call `SendTaskSuccess` or `SendTaskFailure`. **If nobody responds in 5 minutes, no alert is sent** — fail-closed.
3. **BLOCKED** → LogAndStop. Guardrails detected unsafe content. Audit row written, no SMS, no retry. A human dispatcher can issue guidance manually.

**Drop in (the line that lands with judges):**
> "We don't fail-open. The default behavior on uncertainty, on Guardrails block, on Bedrock outage, on dispatcher timeout — all of them route to 'no alert sent.' A late alert is bad. A wrong alert can get someone killed."

**Fallback:** If the Step Functions console UI is slow to load, the same routing is visible in `infrastructure/stacks/safety_stack.py` lines 217-303 — the `_provision_state_machine` method is heavily commented.

## Beat 5 — Verify the chain live (3:30 → 4:15) "prove it"

This is the credibility beat. Skip if running over time.

**Open the terminal.**

**Say:**
> "The audit chain is verifiable. There's a `verify_chain` function — paginates the GSI for a fire, replays every row, recomputes each hash, and checks that every `prev_hash` matches. It returns `True` only if the chain is intact end-to-end."

**Run:**
```bash
python3 -c "
import os
os.environ['WW_AUDIT_TABLE'] = 'wildfire-watch-audit'
from functions.alert.audit import verify_chain
print(verify_chain('<fire_id from beat 3>'))
"
```

**Audience sees:** `True`.

**Drop in:**
> "If anyone tampers with a row in DynamoDB — changes the SMS text, deletes a guardrails_outcome — this returns False. We can build that into a CloudWatch alarm or a periodic Lambda."

**Fallback:** If the script fails (env var missing, AWS creds expired), pivot to "the function is 25 lines, in `functions/alert/audit.py:142`" and show the code. The point is conceptual integrity, not a live shell demo.

## Things to NOT say (will trigger painful follow-ups)

- ❌ "Real-time SMS to thousands of residents" — workshop account SMS is blocked. Say "designed to" or "the call is wired."
- ❌ "Production-ready" — it's a hackathon prototype on an ephemeral workshop account. Say "the architecture is production-shaped."
- ❌ "QLDB" — we don't use it. AWS no longer accepts new QLDB ledgers. We use DynamoDB hash chain.
- ❌ "Pinpoint" — same. Workshop SCP blocks it. We use `sns.publish(PhoneNumber=...)` direct.
- ❌ "Custom domain" — workshop account, no Route53 setup. Demo URL is `*.amplifyapp.com` and that's correct.
- ❌ "Our model achieves X% accuracy" — we don't have held-out validation numbers; the SageMaker endpoint runs on synthetic features. Talk about the framework, not metrics we can't defend.
- ❌ "QLDB-equivalent immutability" — say "tamper-evident hash chain" instead. Don't claim properties we haven't formally proven.
